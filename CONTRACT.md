> Этот файл — не пожелание, а правила. Их цель одна: **FamilyHub не влияет на основное приложение OYNA и не ломает его логику.**

**Статус:** активен с момента первого коммита в репо.
**Владелец контракта:** Madiyar (CEO). **Ревью решений:** Айдар (Tab 1). **Имплементация:** Архи / Claude Code (Tab 2).

## 0. Что такое FamilyHub

Ambient / display-поверхность OYNA для планшета → настенной панели → hub-устройства.
Один профиль ребёнка, один Supabase, одна auth — общие с телефоном (STRAT-008).
FamilyHub — это **отдельный репозиторий и отдельная сборка**, но **read-mostly клиент** тех же данных.

FamilyHub НЕ является: источником схемы данных, местом монетизации, вторым auth-контуром, форком продуктовой логики телефона.

---

## 1. Ноль миграций из этого репо — ГЛАВНОЕ ПРАВИЛО

«Сломать приложение» происходит на уровне **данных**, а не кода.

- FamilyHub **никогда** не создаёт, не применяет и не откатывает миграции — ни в DEV, ни в PROD.
- Любая нужда в изменении схемы идёт через **основной репо OYNA** по стандартному флоу: DEV → верификация → PROD, одна гипотеза за раз, явное «ок выполняй».
- Если фиче FamilyHub нужна новая колонка/таблица/вью — это тикет в основном репо, не здесь.

Это правило одно закрывает большую часть риска. Не нарушать.

---

## 2. Критические объекты БД — не трогать

- `quests` — 108 строк, 3 зависимых FK. **Никогда не дропать, не переименовывать.** Канон новой схемы — `quests_v3` (параллельная таблица).
- `parents` — ключ `auth_uid` (**НЕ** `auth_user_id`). Читать по правильному ключу.
- FamilyHub читает данные **только через существующие интерфейсы** и **уважает RLS**. Никакой новой логики поверх чужих таблиц, никаких Hub-специфичных мутаций в общих таблицах.

Если Hub-фиче нужно что-то писать — писать можно только в объекты, созданные под Hub через основной репо (правило 1), и только с явного согласования.

---

## 3. Общий код — только read-only, через пакет. Основной `lib/` не редактируется

- FamilyHub **не импортирует и не правит** `lib/` основного проекта. Никогда.
- Переиспользуемое (data-модели, дизайн-токены, обёртка Supabase-клиента) выносится в **опубликованный Dart-пакет**, который *оба* проекта тянут как зависимость с зафиксированной версией.
- Пока пакета нет — FamilyHub держит **свои тонкие копии** нужных моделей. Копия лучше, чем импорт с правкой.
- Правка основного репо в рамках Hub-работы — **запрещена**. Изменение общего контракта данных инициируется из основного репо, версионируется в пакете, затем подтягивается сюда.

**Принятая цена (проговорена один раз):** дрейф общих моделей синхронизируется вручную через версию пакета. Это налог отдельного репо. Пакет его минимизирует — не пропускать.

---

## 4. Отдельная сборочная поверхность

- Свой bundle id: `com.oyna.hub` (не пересекается с основным приложением).
- Своё signing, свой build-пайплайн, свои смоук-аккаунты.
- Форм-фактор: планшет / landscape-first, ambient always-on режим.
- **Монетизация остаётся на телефоне.** RevenueCat в FamilyHub не тащим. Hub — это display, не касса.

---

## 5. Секреты (ARCH-RULE-001)

- Все ключи — в env / Supabase Edge secrets FamilyHub, **не в репо**. Никаких ключей в коде/конфигах под git.
- FamilyHub работает на **anon key + RLS** для чтения.
- Запрос `service_role` из Hub — **стоп-сигнал.** Остановись, позови Айдара.
- Правило «никогда не показывать содержимое `.env*`» действует и здесь — только маскированные команды.

---

## 6. Auth

- FamilyHub логинится как **родитель** (или device-scoped сессия под родителем) и читает данные своей семьи.
- **Не** заводит параллельный auth-флоу, который может разойтись с телефоном.
- Сессия ребёнка на Hub (если появится) наследует текущую модель безопасности основного приложения; до pre-v1.1 security-аудита (STRAT-009) — минимальная поверхность.

---

## 7. Гейты (что можно сейчас, что после BET-001)

- **Сейчас (software dogfood):** запуск на планшете в hub-layout у семьи Madiyar. Это product-discovery форм-фактора — **не** прогресс по BET-001 (одна семья, F&F, качественный сигнал).
- **После BET-001 (D30 ≥ 30% на 200+ non-F&F семьях):** железо — настенная панель, kiosk-lockdown (MDM/Knox), кастомный hub.

Dogfood на Hub не маскируется под прогресс беты. Разводим в отчётности.

---

## 8. Definition of «не сломал приложение» (чек перед любым мержем)

Мерж в FamilyHub допустим только если ВСЁ верно:

- [ ] Ни одной миграции в этом PR.
- [ ] Ни одной записи в общие/критические таблицы (`quests`, `parents`, ...).
- [ ] Основной репо OYNA не изменён этим PR.
- [ ] Нет `service_role`; работает на anon + RLS.
- [ ] Нет секретов в git.
- [ ] Смоук основного приложения на телефоне зелёный (телефон не задет).

Любой невыполненный пункт → стоп, эскалация Айдару.
